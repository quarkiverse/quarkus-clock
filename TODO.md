## Code style

Quarkiverse extensions follow the same formatting rules as Quarkus core. The [Checklist for new projects](https://github.com/quarkiverse/quarkiverse/wiki/checklistfornewprojects) and [Quarkus CONTRIBUTING — IDE config and code style](https://github.com/quarkusio/quarkus/blob/main/CONTRIBUTING.md#ide-config-and-code-style) describe the upstream convention; **`quarkus-clock` inherits enforcement from `io.quarkiverse:quarkiverse-parent`**, not from files in this repo.

### How it is enforced here

Parent POM (`quarkiverse-parent` v20) configures:

| Tool | Purpose |
|------|---------|
| `formatter-maven-plugin` | Eclipse formatter using `io.quarkus:quarkus-ide-config` → `eclipse-format.xml` |
| `impsort-maven-plugin` | Import order using `eclipse.importorder` from the same JAR |

On a normal build (without `-Dno-format`), `process-sources` **auto-formats** Java sources and sorts imports.

**Format only (no full build):**

```bash
mvn process-sources
```

**Full build with tests:**

```bash
mvn clean install
```

**Validate formatting without rewriting files** (when `no-format` property is *not* set, `validate` phase runs formatter + impsort checks):

```bash
mvn validate
```

### IDE setup (for contributors)

Do **not** duplicate the long Eclipse/IDEA sections from Quarkus `CONTRIBUTING.md` in this repo. In `CONTRIBUTING.md`, link to Quarkus and add these project-specific notes:

1. Formatter config lives in the Maven artifact, not in this repository:
   ```bash
   jar tf ~/.m2/repository/io/quarkus/quarkus-ide-config/3.0.0.Final/quarkus-ide-config-3.0.0.Final.jar
   # eclipse-format.xml, eclipse.importorder
   ```
   Run `mvn process-sources` once so the JAR is downloaded, then point your IDE at those extracted paths (or extract them to a local `ide-config/` folder if the team prefers).

2. **IntelliJ IDEA** — install [Adapter for Eclipse Code Formatter](https://plugins.jetbrains.com/plugin/6546-eclipse-code-formatter), use `eclipse-format.xml`, enable optimize imports from `eclipse.importorder`, set wildcard import threshold to `999` (see [Quarkus IDEA setup](https://github.com/quarkusio/quarkus/blob/main/CONTRIBUTING.md#idea-setup)).

3. **Eclipse** — import `eclipse-format.xml` and `eclipse.importorder` under Java → Code Style (see [Quarkus Eclipse setup](https://github.com/quarkusio/quarkus/blob/main/CONTRIBUTING.md#eclipse-setup)).

4. **Line endings** — LF (configured in parent formatter plugin).

### Coding guidelines to mention in `CONTRIBUTING.md`

Summarize from [Quarkus coding guidelines](https://github.com/quarkusio/quarkus/blob/main/CONTRIBUTING.md#coding-guidelines), adapted for an extension:

- No `@author` Javadoc tags (use Git history)
- Meaningful, atomic commits; squash fixups before merge
- Prefer limiting lambdas/streams in **runtime** code where footprint matters
- No `@author`-style license headers required in every file (root `LICENSE` is enough unless you add headers later)
- Update tests and Antora docs when changing behavior or configuration

### CI gap (fix recommended)

`.github/workflows/build.yml` currently runs:

```bash
mvn -B clean install -Dno-format
```

`-Dno-format` **disables** both auto-format and format validation from `quarkiverse-parent`. The [checklist wiki](https://github.com/quarkiverse/quarkiverse/wiki/checklistfornewprojects) expects CI to fail when sources are not formatted.

- [ ] Remove `-Dno-format` from CI (or run a separate `mvn validate` job without it)
- [ ] Run `mvn process-sources` locally and commit formatted sources before enabling strict CI
- [ ] Document in `CONTRIBUTING.md`: run `mvn process-sources` before every PR
